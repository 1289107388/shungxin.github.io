## Hard Constraints
- Comment system must use Giscus for real-mode storage
- Cross-domain communication between main site and subdomain must use postMessage protocol
- Like functionality must use Supabase for real storage
- Frontend must call Edge Function API (POST /functions/v1/like-toggle) for like operations; direct database writes are forbidden
- Giscus Webhook must be configured with Payload URL: https://qlhfyawbyedhqokivezn.supabase.co/functions/v1/comment-api/sync-giscus, Content type: application/json, and Events: Discussions + Discussions comment
- Supabase configuration (url + anonKey) must be exposed to global scope via window.SUPABASE_CONFIG to ensure cross-IIFE access
- All Supabase Edge Function calls must include the Authorization: Bearer header
- Database 'comments' table 'source' column has CHECK constraint allowing only 'local' and 'giscus' values
- admin-api Edge Function must have 'Verify JWT' gateway validation disabled to allow custom HMAC token authentication
- auth-api Edge Function must have 'Verify JWT' gateway validation disabled to allow login functionality
- upload-image Edge Function must have 'Verify JWT' gateway validation disabled to allow custom HMAC token authentication
- Batch operations have a single batch limit of 200 items; exceeding returns 400
- Users cannot perform batch operations on themselves; their ID is automatically skipped and returned in 'skipped[]'
- User deletion must first revoke all their sessions to prevent login with old tokens
- Image uploads must be stored in Supabase Storage 'gallery-images' bucket (max 10MB per image)
- User uploads require HMAC token authentication and have a daily limit of 5 images per user

## Pitfalls & Lessons Learned

### 1. Edge Function public endpoints still need `Authorization: Bearer <anon_key>`
- Supabase gateway has **Verify JWT enabled by default** for all Edge Functions.
- Only sending `apikey` results in HTTP 401.
- Public functions like `public-gallery` work because they send **both** `apikey` and `Authorization: Bearer <anon_key>`.
- Reference fix: `js/collections.js` `getHeaders()` must include `Authorization: 'Bearer ' + ANON_KEY`.

### 2. Always add cache-busting query string after editing JS
- Browser aggressively caches static JS files.
- After any JS fix, bump the script tag: `<script src="js/xxx.js?v=2"></script>`.
- Applies especially to `config.js`, `siteSettings.js`, `collections.js`.

### 3. Database type mismatches to watch for
- `public.users.id` is **BIGINT**, `auth.uid()` returns **UUID**. Never use `users.id = auth.uid()` in RLS policies.
- `image_views.image_id` and `image_likes.image_id` are text/varchar; `gallery_images.id` is bigint. Cast in joins/views: `iv.image_id::bigint = gi.id`.
- `collections.created_by` must be **BIGINT** to match `users.id`.

### 4. service_role bypasses RLS; custom HMAC auth does not map to Supabase Auth
- Admin/user management uses custom HMAC tokens verified in Edge Functions.
- `auth.uid()` is essentially useless in this project; admin checks are done via `verifyToken()` + `users.id = payload.sub`.

### 5. Migration deployment workflow
- `supabase db push --project-ref <ref>` is invalid.
- Correct: `supabase link --project-ref <ref>` then `supabase db push --yes`.

### 6. JWT payload parsing
- Payload is the **middle** segment: `JSON.parse(atob(parts[1]))`. `parts[0]` is the header.

### 7. Windows PowerShell command chaining
- Use `;` not `&&` for sequential commands in PowerShell.

### 8. Key file map for collections feature
- Frontend: `js/collections.js`, `js/upload.js`, `js/auth.js`.
- Backend: `supabase/functions/collections-api/index.ts`, `supabase/functions/admin-api/index.ts`.
- Schema: `supabase/migrations/20260628000000_create_collections.sql`.
- Pages: `index.html` (public), `paid.html` (paid), `collection.html` (detail), `admin.html` (management).
